# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

Frontend (run from repo root, package manager is `pnpm`):

- `pnpm dev` ‚Äî Vite dev server on port 3000
- `pnpm build` ‚Äî production build
- `pnpm preview` ‚Äî preview the production build
- `pnpm test` ‚Äî run Vitest once (`vitest run`). For watch mode use `pnpm exec vitest`; for a single file `pnpm exec vitest path/to/file.test.ts`
- `pnpm lint` ‚Äî ESLint
- `pnpm format` ‚Äî Prettier write
- `pnpm typecheck` ‚Äî `tsc --noEmit`

Supabase local dev (the `supabase` CLI is a devDependency, so use `pnpm exec supabase ...`):

- `pnpm exec supabase start` ‚Äî bring up the local stack (Postgres, Auth, Studio, Mailpit, etc.) using `supabase/supabase/config.toml`. Run from repo root or from inside `supabase/` ‚Äî the CLI walks up looking for a `supabase/` config dir.
- `pnpm exec supabase stop` / `pnpm exec supabase status`
- `pnpm exec supabase db reset` ‚Äî drop, re-run migrations from `supabase/supabase/migrations/`, then apply `supabase/supabase/seed.sql`
- Default ports: API/Kong `54321`, Postgres `54322`, Studio `54323`, **Mailpit web UI `54324`**, Analytics `54327`. All emails GoTrue sends (signup, password recovery, magic links, email change) land in Mailpit at http://localhost:54324.
- Frontend env vars (`SUPABASE_URL`, `SUPABASE_ANON_KEY`) come from `supabase status` output.

The `supabase/` directory **also** contains a vendored copy of the upstream self-hosted docker-compose tree (`supabase/docker-compose.yml`, `supabase/dev/`, `supabase/.env`, `supabase/utils/*.sh`). That is **not** what the dev workflow uses ‚Äî it's a separate self-host reference, kept here for production deployment scenarios. Don't run it alongside `supabase start`; the port maps will clash.

## Architecture

**TanStack Start** (SSR React framework built on Vite + Nitro) with **self-hosted Supabase** as the backend. There is no separate backend service in this repo ‚Äî server-side logic lives inside the TanStack Start app as server functions that call Supabase.

### Frontend (`src/`)

- **Routing**: file-based via `@tanstack/react-router`. Route files live in `src/routes/`; `src/routeTree.gen.ts` is auto-generated by the router plugin ‚Äî never edit it. Layout routes use a `route.tsx` file (e.g. `routes/auth/route.tsx` wraps everything under `/auth`). `__root.tsx` is the root layout and owns `<ThemeProvider>` and devtools.
- **Router bootstrap**: `src/router.tsx` builds a `QueryClient` and wires `setupRouterSsrQueryIntegration`, so React Query state is hydrated across SSR boundaries. Route loaders/components can use the `queryClient` from route context.
- **Server functions**: created with `createServerFn` from `@tanstack/react-start` (see `src/lib/server-functions/auth.ts`). They run on the server, validate input via `.inputValidator(zodSchema)`, and are called from the client like async functions (`useMutation({ mutationFn: loginFunc })` passes `{ data: value }`).
- **Supabase client**: `src/lib/utils/supabase.ts` builds a `createServerClient` (`@supabase/ssr`) that reads/writes cookies through TanStack Start's `getCookies` / `setCookie` from `@tanstack/react-start/server`. Auth is session-cookie based; do not introduce a browser-side `createBrowserClient` unless you have a clear reason ‚Äî server functions are the integration point. Requires `SUPABASE_URL` and `SUPABASE_ANON_KEY` env vars.
- **Forms**: `@tanstack/react-form-start` with Zod validators in `src/lib/validators/`. Pattern: declare a validator once and pass it as both `onMount` and `onChange` to keep submit state in sync.
- **Generated DB types**: `src/lib/types/supabase.ts` is the generated Supabase schema ‚Äî treat as generated, regenerate via Supabase CLI rather than editing.
- **Theme**: `ThemeProvider` (`src/components/theme-provider.tsx`) stores `light`/`dark`/`system` in `localStorage` and injects an inline `<ScriptOnce>` (`getThemeScript`) to set the class before hydration to prevent flash.
- **UI**: shadcn/ui configured via `components.json` with the `base-luma` style, `neutral` base color, and **HugeIcons** as the icon library (`@hugeicons/react` + `@hugeicons/core-free-icons`) ‚Äî not Lucide. Add components with `npx shadcn@latest add <name>`; they land in `src/components/ui/`. The `cn` util lives at `@/lib/utils/cn` (not `@/lib/utils`).

### Path aliases (tsconfig)

- `@/*` ‚Üí `src/*`
- `@components/*` ‚Üí `src/components/*`
- `@lib/*` ‚Üí `src/lib/*`

Both `@/components/...` and `@components/...` resolve. Existing code mixes them ‚Äî prefer the scoped aliases (`@components`, `@lib`) for new code in `src/`.

### Supabase directory (`supabase/`)

Two layouts live side-by-side here ‚Äî don't confuse them:

1. **`supabase/supabase/` ‚Äî the live local-dev config** managed by the Supabase CLI.
   - `config.toml` ‚Äî single source of truth for ports, services, auth providers, mailer, etc. The `[inbucket]` section is the mailer; the CLI keeps the legacy name but the actual image is Mailpit.
   - `migrations/` ‚Äî SQL migrations applied by `supabase db reset` / `supabase db push`
   - `seed.sql` ‚Äî dev seed
   - `snippets/` ‚Äî saved Studio SQL snippets

2. **`supabase/` (top level) ‚Äî vendored self-hosted docker-compose**, mirrored from `supabase/supabase` upstream as a reference for production self-hosting. Files: `docker-compose.yml`, `dev/docker-compose.dev.yml`, `utils/*.sh`, `tests/*.sh`, `CHANGELOG.md`, `versions.md`. Update by syncing from upstream rather than hand-editing. `supabase/volumes/` is gitignored and only created if you actually run this stack.

## Conventions

### ESLint vs Prettier conflict (important)

The two tools disagree and **ESLint is the source of truth** ‚Äî actual committed code follows ESLint:

| | Prettier (`.prettierrc`) | ESLint (`eslint.config.js`) | Code follows |
|---|---|---|---|
| Semicolons | `semi: false` | `@stylistic/semi: always` | **semicolons** |
| JS string quotes | `singleQuote: false` (double) | `@stylistic/quotes: single` | **single** |
| JSX attribute quotes | (default double) | `@stylistic/jsx-quotes: prefer-double` | **double** |

Do **not** run `pnpm format` blindly across the codebase ‚Äî it will introduce lint errors. Use `pnpm lint --fix` for autofixable style. If you must run Prettier, restrict it to files you know are safe (e.g. JSON, CSS ‚Äî the prettier plugin in use is `prettier-plugin-tailwindcss` for class sorting).

Other ESLint rules worth knowing:
- `@stylistic/jsx-self-closing-comp` is enforced for both components and html ‚Äî write `<div />` not `<div></div>` for empty elements.

### TypeScript

- Strict mode plus `noUnusedLocals`, `noUnusedParameters`, `noFallthroughCasesInSwitch`, `noUncheckedSideEffectImports`, `verbatimModuleSyntax` ‚Äî use `import type` for type-only imports.
- React 19 + the new JSX transform (`jsx: react-jsx`); don't import `React` just for JSX.
